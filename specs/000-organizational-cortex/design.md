# Design — 000 Organizational Cortex MVP

## Design Intent

Create an executive-grade interactive narrative rather than a conventional analytics dashboard. The MVP should make the AI-native operating model tangible while remaining honest about synthetic data and unvalidated hypotheses.

## Experience Architecture

```text
Executive Landing
  -> Organizational Cortex
     -> Student Retention Simulation
     -> Teacher Intelligence Simulation
     -> Agentic Build Engine
  -> 30/60/90-Day Roadmap
  -> Transformation Blueprint Preview
```

## Proposed Technical Shape

The implementation stack will be selected only after approval and a current-documentation checkpoint. The preferred shape is:

- React-based web application
- TypeScript
- Static synthetic datasets
- Component-driven UI
- Lightweight client-side state
- Automated unit and interaction tests
- Static deployment capability

No backend is required for the first MVP unless an approved implementation finding demonstrates that it is necessary.

## Domain Model

### Interaction

A class, lesson action, support interaction, teacher behavior, pull request, evaluation, or business event.

### Signal

A structured observation derived from an interaction. Signals must include source, timestamp, confidence, and domain.

### Evaluation

A deterministic or model-assisted interpretation of one or more signals. Evaluations must distinguish facts from inference.

### Organizational Memory

A reusable pattern, decision, learned behavior, or validated relationship that can influence future recommendations.

### Recommendation

A proposed action with rationale, confidence, expected outcome, and a human ownership boundary.

### Outcome

A measurable result used to validate or reject the recommendation and update organizational memory.

## Screens

### 1. Executive Landing

- Transformation thesis
- Current-state fragmentation versus future-state learning loop
- Two transformation dimensions: AI-Native Product and AI-Native Company
- Priority pain points: early churn and teacher variability
- Entry points into the simulations

### 2. Organizational Cortex

- Closed-loop visualization
- Domain nodes for students, teachers, curriculum, support, sales, finance, and engineering
- Live-looking but deterministic synthetic signal feed
- Explanation of how knowledge compounds

### 3. Student Digital Twin

- Synthetic learner identity
- Goal, proficiency, schedule, motivation, confidence, recurring errors, and engagement
- Risk indicator with evidence panel
- Recommended intervention and expected outcome
- Clear uncertainty and human-review affordance

### 4. Teacher Intelligence

- Synthetic cohort distribution
- Top-performer behavioral patterns
- Selected teacher strengths, development areas, recommendations, acceptance, and measured impact
- Guardrail explaining that behavior-outcome relationships require controlled validation

### 5. Agentic Build Engine

- Feature lifecycle
- Active feature and scope
- Spec, approval, implementation, verification, review, and production gates
- Evidence generated by this repository

### 6. 30/60/90-Day Roadmap

- Days 0–30: domain model, event taxonomy, telemetry, agentic SDLC, governance
- Days 31–60: student retention and teacher intelligence pilots
- Days 61–90: validated interventions, platform decisions, enterprise integration sequencing, Board metrics

## Data Design

All initial data will be local, deterministic, and synthetic. Data objects should be typed and shaped as if they were API responses so the mock layer can later be replaced.

Every synthetic entity must include:

```text
isSynthetic: true
scenarioVersion
sourceDescription
```

## Explainability Pattern

Every AI-like output must expose four layers:

1. Observed evidence
2. Inference
3. Recommendation
4. Outcome or validation status

The UI must not present confidence as certainty.

## Visual Direction

- Cinematic but restrained
- Executive, premium, and data-driven
- High contrast and generous whitespace
- Narrative transitions rather than excessive cards
- Motion used to explain system behavior, not decorate it
- No copied Open English proprietary visual assets

## Architecture Boundaries

### Allowed in MVP

- Static frontend
- Local fixtures
- Deterministic calculations
- Simulated agent and evaluation states
- Reusable design system primitives

### Deferred

- Real LLM calls
- Vector databases or knowledge graphs
- Authentication
- Persistent storage
- Enterprise integrations
- Production AWS architecture
- Personally identifiable information

## Verification Strategy

The approved implementation spec must define exact commands for:

- type checking
- linting
- unit tests
- interaction tests
- production build
- accessibility smoke checks
- responsive viewport checks

## Risks and Mitigations

- Risk: The prototype looks like unsupported company analysis.
  - Mitigation: Use hypothesis language and synthetic labels.
- Risk: The scope expands into a full LMS.
  - Mitigation: Keep three simulations and one executive narrative only.
- Risk: Visual polish outruns product clarity.
  - Mitigation: Acceptance tests focus on the decision narrative.
- Risk: AI outputs appear magical.
  - Mitigation: Enforce evidence, inference, recommendation, and outcome separation.
- Risk: Interview time is consumed by architecture detail.
  - Mitigation: Progressive disclosure, with an executive path under five minutes.

## Decision Required Before Implementation

Approve or amend:

- the three-domain MVP scope;
- static synthetic data as the initial data strategy;
- executive narrative as the primary product surface;
- no backend and no external AI calls for the first increment;
- cinematic, restrained visual direction.
